```

 │

 │       ├── _status.json         # Agent 状态快照
 │       └── worklogs/            # 会话内工作日志（各 Agent 自行创建）
 │
 └── logs/
     └── agentd.log              # 调度器运行日志
```

## 二、bridges/ 的设计定位

### 2.1 为什么要有 bridges/

agentd 的职责是「发通知」，不负责「产生回复」。

每个 AI Agent（WorkBuddy、Claude Code 等）需要一个**驻留进程**来：
1. 监听 HTTP 端口 → 收到 agentd 的 POST 通知
2. 调用对应的模型 API → 生成回复
3. 写回 session.md

这个驻留进程就是 **bridge**。`bridges/<Agent名>/` 就是存放 bridge 源码和配置的地方。

### 2.2 为什么这么轻

bridge 不是工程项目，它只有三样东西：

| 文件 | 是否必选 | 说明 |
|------|---------|------|
| `<agent>_bridge.py` | 是 | HTTP 服务，收通知→调 API→写回 session.md |
| `start.bat` (或 .sh) | 是 | 启动入口，设置环境变量后执行 bridge |
| `memory/` | 否（推荐） | 跨会话长期记忆（见第3章） |

不需要 config/、tests/、README.md 这些工程化目录。一个 bridge 的核心逻辑就是几件事：
- 一个 HTTP handler（/chat 端点）
- 一个 API 调用（DeepSeek / Claude / 其他）
- 一个文件追加操作

### 2.3 谁来创建 bridges/<Agent名>/

**Agent 自己创建**，用户只需把指令发给 Agent。

现有的 README "获取 Agent 配置" 提示词已经告诉 Agent：
1. 启动 HTTP 服务（需持续运行）
2. 注册到 agents.json

需要新增一条：**3. 把桥接脚本保存到 `bridges/<你的名字>/` 目录下，并创建 `memory/` 目录**。

这样 Agent 下次启动时直接从 bridges/ 找自己的脚本，不需要用户记忆它放哪了。

### 2.4 扩展性

未来加入新 Agent（如 Generic Agent、某垂直领域 Agent）：

```
bridges/
├── workbuddy/
│   ├── workbuddy_bridge.py
│   ├── start.bat
│   └── memory/
├── claude-code/
│   ├── claude_code_bridge.py
│   ├── start.sh
│   └── memory/
└── generic/                  ← 新 Agent 进来自己创建
    ├── generic_bridge.py
    ├── start.bat
    └── memory/
```

每个 Agent 完全自治，互不依赖。

## 三、memory/ 跨会话持久记忆

### 3.1 为什么需要

现状：

| 维度 | 存储位置 | 作用域 | 能延续到下次会话？ |
|------|---------|--------|-----------------|
| 小组协作 | `session.md` | 本次会话 | ❌ |
| 个人工作记录 | `worklogs/` | 本次会话 | ❌ |

每个新会话，Agent 对上一轮项目的认知、自己的角色定位、踩过的坑，**全部归零**。

memory/ 解决了这个问题：**Agent 自己的跨会话长期记忆**。

### 3.2 设计

```
bridges/WorkBuddy/memory/
└── session.md           # 追加式记忆文件，和 session.md 格式一致
```

- **格式**：和 session.md 完全一致，Agent 自身追加写入，不修改已有内容
- **内容**：Agent 自己觉得"以后会用到"的东西——项目上下文、踩坑记录、偏好结论
- **生命周期**：跨会话持久，Agent 启动时主动读取注入 system prompt

### 3.3 写入时机

每次 bridge 完成任务、生成回复前，`# 可选：如果本次回复包含值得记忆的内容，追加到 memory/session.md`。

什么算"值得记忆"：
- 团队做出的关键架构决策
- 自己发现的最佳实践 / 易错点
- 对某个 Agent 协作风格的观察
- 技术栈选择理由

### 3.4 注入方式

bridge 启动时 / 每次收到通知时，读取 `memory/session.md` 内容，作为 system prompt 的一部分：

```python
# 在 bridge 的 system prompt 尾部拼接记忆
memory_content = read_file("memory/session.md")
system_prompt = f"""
你是一个 AI Agent...
...
## 你的长期记忆（跨会话持久）
{memory_content}
"""
```

这样 Agent 在回答时就有历史上下文。

### 3.5 与 worklogs/ 的区别

| | worklogs/ | memory/ |
|---|---|---|
| 存放位置 | `sessions/<会话>/worklogs/` | `bridges/<Agent名>/memory/` |
| 作用域 | 单次会话 | 跨会话 |
| 用途 | 当前任务的详细工作记录（代码、日志、调试输出） | 长期有价值的经验和知识沉淀 |
| 谁写 | Agent 自己 | Agent 自己 |
| 谁读 | 其他 Agent（通过 session.md 引用路径） | 只有自己（注入 system prompt） |

两者互补，worklogs 解决"细节写不下"，memory 解决"记忆留不住"。

## 四、架构关系

```
用户启动流程：

1. 用户启动 Agent bridge 进程（每个 Agent 一个终端）
   └─ workbuddy → python bridges/workbuddy/workbuddy_bridge.py
   └─ claude    → python bridges/claude/claude_bridge.py

2. 用户运行 python man.py
   └─ 启动 agentd.py（调度器 :3001）
   └─ 启动 ui.py（用户输入窗口）
   └─ 读取 agents.json，通知各 Agent
   └─ 等待签到 → 进入监控循环

运行时数据流：

  session.md ←── Agent bridge 追加回复
      ↑                   ↑
      │             调 API + 读 memory/
      │                   │
 agentd.py ──HTTP POST──→ Agent bridge（:3010 / :3011 / ...）
（监控变化、解析###、通知对应Agent）
```
